Orchestration

provides several orchestration options so that you can deploy wherever it suits your needs; locally, on-premise, in the cloud, or in a hybrid environment.

Standalone Mode

This is a simple cluster manager that creates a cluster on your local machine. Use standalone mode to evaluate products, or during development and testing.

GigaOps Stack

The GigaOps stack is comprised of the following:

  • gsctl - a standalone CLI tool that you can use to create, provision, and install clusters in the cloud or on premise.
  • Ops Manager - the native administration and monitoring tool, which is web based and runs automatically when services are deployed.
  • Monitoring stack - based on InfluxDB, Grafana, and Telegraf for monitoring cluster activity and metrics using dashboards
  • ElastiGrid or Kubernetes orchestration

ElastiGrid

Cloud environments are scalable, reliable, and highly available, as well as highly cost effective. You can deploy in the cloud, enabling you to easily test, buy and deploy without having to commit to owning and maintaining an on-premise infrastructure. ElastiGrid is ' orchestration option for cloud, on-premise, or hybrid environments that provides automated deployment, scaling, and management of virtualized and containerized applications.

ElastiGrid has the following benefits:

  • Ability to run containers and legacy applications on the same infrastructure
  • Automatic handling of application and node failure due to its distributed and resilient architecture
  • Multi-region federation that enables linking multiple clusters
  • Scalability for changing workload requirements

 

Kubernetes

KubeGrid

products can be deployed in Kubernetes using the KubeGrid orchestration option. Kubernetes can be used in any type of environment (on-premise, cloud, or hybrid) and is supported by every major cloud provider, including public cloud platforms like AWS, Microsoft Azure, and GCP. Benefits of using KubeGrid include:

  • The ability to deploy -based applications in whatever environment best suits the business needs of the enterprise.
  • Kubernetes synergizes with products, simplifying the operationalizing of machine learning and transactional processing at scale.
  • utilizes key features of Kubernetes, such as cloud-native orchestration automation with self-healing, cooperative multi-tenancy, and RBAC authorization.
  • Auto-deployment of data services and deep learning and machine learning frameworks, such as Apache Spark jobs, stateful services (Processing Units), as well as the Apache Zeppelin web notebook.

Service Grid

The service grid is an on-premise system for deploying and managing virtualized applications. This orchestration method is responsible for materializing the configuration of the Processing Units, provisioning the instances to the runtime infrastructure, and making sure they continue to run properly over time. Core components of the Service Grid include the Grid Service Manager, Grid Service Agent, and the Manager.

Docker

products can be deployed on the Service Grid using Docker images that are available online through Docker Hub. Using Docker simplifies environment setup on-premise as well as moving to the cloud, because all the major cloud vendors support Docker. The following Docker images are available on Docker Hub (each Docker image provides instructions on how to use it):

  • gigaspaces/insightedge - Contains the open source edition.
  • gigaspaces/insightedge-enterprise - Contains the Enterprise edition (requires a license key).
  • gigaspaces/xap - Contains the XAP open source edition.
  • gigaspaces/xap-enterprise - Contains the XAP edition (requires a license key).

 

 

Orchestration Option Comparison

You can compare the following parameters between the orchestration options, in order to determine which one best suits your needs:

  • Zero configuration required - Can you deploy and run on demand, or easily on existing machines?
  • Auto provisioning - Can you start a cluster from zero?
  • Auto-scaling - Does the environment automatically add and remove nodes as needed?
  • Performance penalty - Does communication between containers affect the performance (speed) of the application?
  • Ease of use - How easy is it to develop applications on top of ?
  • Support for product evolution - Can you perform a gradual and controlled upgrade process for the product version and/or processing unit services?
  • Supported environments - Can your cluster run across regions and even cloud providers?
  • Supported processes - What type of processes are supported (container or native process)?
