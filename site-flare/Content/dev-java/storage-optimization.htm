Storage Optimization

Storage Optimization is a feature of that reduces the memory footprint (RAM) of the the properties of a Space Object, with minimal impact on system performance.

How Does Storage Optimization Work?

In order to reduce the system's RAM footprint, storage-optimized fields (Space Object Properties) are ??? deserialized into a BLOB with indexes to each field value ???. This reduces the amount of memory required for the fields. When the data is retrieved, the BLOB indexes are used to access the data.

How Do I Decide Whether to Use Storage Optimization?

The decision whether or not to use Storage Optimization is similar to how you decide whether to index a field.

Storage Optimization is recommended to reduce the memory footprint of a Space Object that has more than 20 fields, is used remotely (not collocated with the relevant code), and is used primarily for read requests. In this situation, there will be little performance overhead.

Storage Optimization is most effective with non-indexed fields.

How Do I Implement Storage Optimization?

Storage Optimization can be implemented using the Ops Manager user interface, or by specifying annotations in the Java code.

Implement Storage Optimization via Ops Manager

In Ops Manager one-click connect, a toggle on the top of the screen lets you specify Storage Optimization for the Space Object:

Clicking on the toggle will select Storage Optimization for every non-indexed property in the object. You can then individually toggle each field to turn off Storage Optimize if desired.

Implement Storage Optimization via the Java API

An example using the Java API is given below.

API for Storage Optimized (StorageType.BINARY)


// Instructs the space to pack all non-indexed properties in a single binary payload.

// Individual properties can exclude/include themselves from the binary payload using @SpacePropertyStorage

@SpaceClassStorage(StorageType.BINARY)

// Optional: format/layout selection

//@SpaceClassStorage(value = StorageType.BINARY, binaryAdapter = DefaultClassBinaryStorageAdapter.class)

public class Demo {

   private int id;

   private String name;

   private String text;


   @SpaceId

   // Indexed properties are stored as OBJECT by default, but can be overridden:

   //@SpacePropertyStorage(StorageType.BINARY)

   public int getId() {

       return id;

   }


   @SpaceIndex

   // Indexed properties are stored as OBJECT by default, but can be overridden:

   //@SpacePropertyStorage(StorageType.BINARY)

   public String getName() {

       return name;

   }


   // Non-indexed properties are stored as BINARY by default, but can be overridden:

   @SpacePropertyStorage(StorageType.OBJECT)

   // Optional: can be used with user-defined property storage adapter - e.g. suppose I've got an incredible zip algorithm:

   //@SpacePropertyStorageAdapter(MyIncredibleZipAdapter.class)

   public String getText() {

       return text;

   }

}
