Service Account Privilege Management

Introduction

already secures user accounts. This is being expanded to now include the securing of service accounts. A service account differs to a user account in that it is software that runs in our Smart Cache or Smart DIH.  The service account has both internal and external applications:

  • Internal Smart DIH: A deployed pipeline. SQL stream or micro-service.

  • External Applications: An application which wants to read from our Smart Cache or Smart DIH, for example a user who wants to write a Java application to read from our Smart Cache or Smart DIH. It does not necessarily need to be deployed on the same Kubernetes cluster.

An external IDP cannot be relied upon as they cannot manage services, only users.

has a mechanism that will manage the service account applications mentioned above.

Implementation

  • Internal Smart DIH: Services that are created from within our Smart Cache or Smart DIH, such as a Pipeline or a Micro-Service, will automatically be registered.  A service account will be created for any new service and this will be transparent to the user.

  • External Applications: There is an interface through which a user can register their application as a service account so is aware of it.

Once the services are defined, roles can be assigned to them. From then, all the privileges management use the RBAC model which replicates what is implemented for user accounts. Due to this, it seamlessly compliments the already available security authorization management.
